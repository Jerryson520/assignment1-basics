# 训练基础设施及其技巧

> 通用训练工作手册。汇总三块可复用的工程实践：训练日志基础设施、实验记录格式、训练 debug 技巧。适用于任意从零搭建的神经网络训练脚本。

---

## 1. 训练日志基础设施

目标：让训练过程**可观测、可复现、可对比**。

### 开关式接入实验追踪
用一个命令行参数（如 `--wandb-project`）作开关——给定则启用实验追踪后端（Weights & Biases / TensorBoard / MLflow 等），否则只打印 console。这样本地快速调试与正式实验共用一套代码。

```python
use_tracker = args.wandb_project is not None      # 开关
if use_tracker:
    import wandb
    wandb.init(project=..., name=..., config=vars(args))   # 一次性存档全部超参
...
start_time = time.time()                          # 墙钟基准（训练循环开始前）
```

### 双横轴记录
每次记录指标时**同时携带两个坐标**：

```python
elapsed = time.time() - start_time
wandb.log({"train/loss":..., "lr":..., "wallclock_sec":elapsed}, step=step)
```

- `step`：梯度步数轴（默认横轴）。
- `wallclock_sec`：墙钟秒数轴（面板里可切换 x 轴）。

同一个 loss 值同时关联「步数」与「时间」两个坐标。

### 应记录的指标
- **train loss**：每隔 `log_interval` 步。
- **valid loss**：每隔 `eval_interval` 步，在 `@torch.no_grad()` + `model.eval()` 下跑若干 batch 求平均，结束后切回 `model.train()`。
- **learning rate**：当前调度器学习率（验证 warmup / decay 是否符合预期）。
- **wallclock_sec**：自训练开始的累计墙钟时间。

### 超参一次性存档
`config=vars(args)` 把全部命令行超参一次性写入该 run，保证每次实验配置可复现，事后能逐项对照。

### 为什么要墙钟时间轴
仅看「loss vs 步数」会掩盖每步计算成本的差异。做架构 ablation（换算子实现、改 batch size、改模型宽度）时，不同配置每步耗时不同，必须用墙钟时间才能公平比较**真实训练效率**。

### 配套基础设施
- **Checkpoint**：定期保存 `model / optimizer / step` 三者的 state_dict，支持 `--resume` 续训。只存 model 无法恢复优化器动量。
- **离线模式**：无网络时设 `WANDB_MODE=offline`，事后 `wandb sync` 补传。
- **输出目录自动创建**：`os.makedirs(checkpoint_dir, exist_ok=True)`。

## 2. 实验记录格式

> 每做一个 ablation 实验追加一行，并附曲线截图（步数横轴 + 墙钟横轴各一张）。

| 日期 | 实验名 | 改动 | 关键超参 | 最终 valid loss | 步数 | 墙钟时间 | 结论 |
|------|--------|------|----------|-----------------|------|----------|------|
| — | baseline | 标准配置 | lr / d_model / 层数 / 头数 / ctx / batch | _待填_ | _待填_ | _待填_ | 基准线 |
|  |  |  |  |  |  |  |  |

字段约定：
- **改动**：本次相对 baseline 改了什么，一句话。
- **关键超参**：lr / 模型宽度 / 深度 / 头数 / 上下文长度 / batch size。
- **步数 + 墙钟时间**：双口径效率，缺一不可。
- **结论**：这个改动好/坏，为什么。
- 每条 ablation **追加一行**，绝不覆盖历史。
- 每行附曲线截图 **两张**：step 横轴 + wall-clock 横轴。
- 「冒烟验证 / 管线连通性测试」单独标注，**不作为正式实验结果**。

## 3. 训练 debug 技巧

1. **先过拟合单个 batch**：把同一个 batch 反复喂给模型，loss 必须能压到接近 0。压不下去 → 模型 / 反传 / optimizer 有 bug，而非数据或学习率问题。最快定位实现错误。

2. **检查 loss 初值是否合理**：分类任务随机初始化时 cross-entropy ≈ `ln(类别数)`。例如词表 10000 → ≈ `ln(10000) ≈ 9.21`。初值偏离太多通常意味着初始化、softmax 数值稳定性或权重未初始化（如 `torch.empty` 直接用导致 nan）有 bug。

3. **打印张量 shape**：注意力、reshape / rearrange、einsum 最易出维度错。可疑处临时 `print(x.shape)` 比读代码快。

4. **监控梯度 / 参数范数**：loss 突然 nan / inf 多半是梯度爆炸。打印梯度裁剪前的全局 L2 norm；必要时调小 lr 或加 warmup。

5. **固定随机性以复现**：debug 时固定 `torch.manual_seed` + `np.random.seed`，让取 batch、初始化、dropout 可复现，才能确认某个改动是否真有效果。

6. **小配置快迭代**：debug 用小模型 + 短上下文 + 小 batch，CPU 秒级一步，验证管线正确后再上 GPU 跑大配置。

7. **先验证管线再追求效果**：新脚本第一次跑务必用极小步数做冒烟测试——确认 loss 在下降、checkpoint 能存能恢复、日志能上报，再开长跑。
